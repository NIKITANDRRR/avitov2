# 1. Ограничения и режим работы

## Критично
- **Не использовать API Avito.**
- **Не использовать прокси на текущем этапе.**
- MVP запускается на **обычном домашнем компьютере** с обычным провайдером.
- Главная цель текущего этапа: **проверить работоспособность пайплайна**, а не масштабировать сбор данных.
- Нужно проектировать код так, чтобы **позже можно было легко добавить прокси**, но сейчас они отключены.

## Режим сбора данных — только low-traffic
Система должна работать максимально осторожно:
- без многопоточности;
- без массового обхода объявлений;
- без агрессивного скрапинга;
- только через **Playwright + Chromium**;
- предпочтительно **headful** режим на этапе PoC;
- с большими случайными задержками между действиями;
- с обязательным кэшированием и дедупликацией URL.

---

# 2. Цель текущего PoC

Нужно реализовать рабочий минимальный пайплайн:

1. Принять на вход список из **3 Avito search URL**
   (регион: Россия, состояние: Новый).
2. Поочерёдно открыть каждую поисковую страницу через Playwright.
3. Сохранить HTML поисковой страницы.
4. Извлечь ограниченное количество ссылок на объявления.
5. Открыть только небольшое число **новых** объявлений.
6. Сохранить HTML карточек.
7. Распарсить данные.
8. Сохранить всё в PostgreSQL.
9. Посчитать простую рыночную оценку.
10. Найти объявления ниже рынка.
11. Отправить уведомление в Telegram.
12. Завершить работу.

---

# 3. Жёсткие лимиты PoC

Соблюдать обязательно:

- максимум **3 поисковых URL**;
- за один запуск открывать:
  - не более **3 страниц поиска**;
  - не более **2–3 новых карточек на один поиск**;
- итого не более **6–9 карточек объявлений за запуск**;
- не открывать повторно объявления, которые уже есть в БД и недавно проверялись;
- не использовать пагинацию глубже 1 страницы;
- не использовать параллельные браузерные вкладки для массового сбора.

# 4. Требования к архитектуре

Нужно реализовать следующие модули:

## 4.1 Collector
Назначение:
- открыть search URL;
- дождаться загрузки страницы;
- сохранить HTML;
- извлечь ссылки на объявления;
- открыть ограниченное число новых объявлений;
- сохранить HTML карточек.

Требования:
- Playwright
- Chromium
- headful по умолчанию
- случайный User-Agent
- случайные задержки:
  - перед открытием страницы
  - между переходами
  - перед чтением контента
- возможность позже подключить proxy config, но сейчас `USE_PROXY = false`

## 4.2 Parser
Назначение:
- парсить HTML поисковой страницы;
- парсить HTML карточки объявления.

Извлекать поля:
- ad_id
- url
- title
- price
- location
- seller_name, если доступно
- condition, если доступно
- publication date/time, если доступно
- search_phrase/search_url
- first_seen_at
- scraped_at

Парсер должен быть устойчивым:
- несколько fallback-селекторов;
- обработка отсутствующих полей;
- без падения всего пайплайна из-за одной карточки.

## 4.3 Storage
Использовать PostgreSQL.

Нужны таблицы минимум:
- `tracked_searches`
- `search_runs`
- `ads`
- `ad_snapshots`
- `notifications_sent`

Требования:
- хранить уникальность по `ad_id` или нормализованному `url`
- хранить сырые HTML-файлы на диске, а в БД — путь к ним
- хранить timestamps
- обеспечить дедупликацию

## 4.4 Price Analyzer
Нужно реализовать простую логику определения цены ниже рынка.

Для MVP:
- считать рынок по накопленным данным для каждого search_url или search_phrase;
- использовать медиану;
- если данных достаточно, можно использовать Q1;
- помечать объявление как `undervalued`, если цена существенно ниже медианы;
- порог вынести в конфиг.

Пример:
- `is_undervalued = price < median_price * undervalue_threshold`

## 4.5 Notifier
Реализовать уведомления в Telegram.

Отправлять сообщение только если:
- объявление ещё не уведомлялось;
- оно определено как `undervalued`.

Формат сообщения:
- заголовок
- цена
- медианная цена по выборке
- отклонение в процентах
- URL
- дата обнаружения

## 4.6 Scheduler
На этапе PoC достаточно простого запуска из CLI.

Нужно:
- команда запуска одного полного цикла;
- отдельная возможность потом подключить cron/systemd/task scheduler;
- в коде предусмотреть случайную задержку перед стартом.

Нужно вынести в конфиг:

---

# 7. Алгоритм одного запуска

Реализовать следующий сценарий:

1. Загрузить конфиг.
2. Подключиться к PostgreSQL.
3. Для каждого search URL:
   - подождать случайную задержку;
   - открыть страницу;
   - сохранить HTML;
   - извлечь объявления из выдачи;
   - отфильтровать уже известные/недавно проверенные;
   - выбрать только 2–3 новых объявления;
4. Для каждого выбранного объявления:
   - подождать случайную задержку;
   - открыть карточку;
   - сохранить HTML;
   - распарсить данные;
   - сохранить/обновить запись в БД;
5. После завершения сбора:
   - пересчитать медианы по накопленным данным;
   - определить undervalued объявления;
   - отправить Telegram уведомления по новым сигналам;
6. Записать результаты в лог;
7. Завершить работу без фоновых процессов.
